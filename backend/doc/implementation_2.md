## Corrections to `implementation.md`'s original phases

Read these before trusting the corresponding phase's own checklist — the phase itself is left untouched (history), but its premise has since changed:

- **Phase 2 — Tenant Provisioning:** `SuperAdmin::AccountsController` no longer creates tenants. Its `#new`/`#create`/`#index`/`#show`/`#edit` actions and views are gone — only `#suspend`/`#reinstate` remain (member actions off a modal on the owning Agency's own show page, not a standalone page). See Phase 19 below — `AgencyConsole::AccountsController` is the only place a tenant `Account` is created now.
- **Phase 5 — Event Approval Workflow (Super Admin gate):** gone entirely. `SuperAdmin::EventReviewsController`, `Event#approval_status`, and the whole pending/approved/rejected review queue were removed (migration `20260719060200_remove_quotation_and_approval_workflow_from_events.rb`). `EventMailer#rejected` and its WhatsApp companion (Phase 13's own item) have no caller left — the mailer file itself is deleted. Event creation is gated on the owning Agency's contract instead (`Event#agency_contract_must_be_active`, Phase 19).
- **Phase 15 — Platform Billing & Invoicing:** its own "Redesign (current)" section (quotation-negotiated per-event pricing) is itself now superseded, not just the original plan-tier version already struck through beneath it. `Quotation`/`QuotationRevision` are dropped tables (migration `20260719060300_drop_quotations_and_quotation_revisions.rb`) — there is no more per-event price negotiation. Pricing is agency-level now (`Agency#billing_cycle`: `per_event` fixed price × pool, or `annual` flat fee) — see Phase 19.
- **Phase 17 — Cross-Tenant SSO (Agency) & Audit Log / Impersonation:** delivered differently than planned, and only partly. The "Agency" SSO half is real and shipped — see Phase 19's `AccountSwitch` item — but as a same-DB single-use token record (`AccountSwitch`, 60s TTL), not the JWT-relay-token-plus-Redis-`jti`-registry design this phase originally specified. **`AuditLogEntry` and Super Admin impersonation were never built** — still a real gap, tracked at the bottom of this file's own Cross-cutting checklist.

Everything else in `implementation.md` (Phases 0, 1, 3, 4, 6, 7, 7.5, 8, 9 [partially — see Phase 20], 10, 11, 13, 14) still accurately describes what's live. Phase 12 (Sponsors/Exhibitors), Phase 16 (Tenant OAuth2 API endpoints — the Doorkeeper *application* auto-creation from Phase 2 still happens, but no actual API endpoint exists to authenticate against), and Phase 18 (Next.js public site — a sibling `frontend/` directory exists but is still the unmodified `create-next-app` scaffold, no real pages) remain genuinely not started.

## Phase 19 — Fixed-Hierarchy Pivot: the Agency Layer

**Goal (confirmed with the user, not originally in `requirement.md`):** *"super admin will create the agency and agency will create the tenants"* — a third tier between Super Admin and tenant Account. Agencies are billed (not individual tenants); an agency's own staff sign into any of its tenants via a single-sign-on switch, using nothing but the ordinary tenant Devise session. This single pivot obsoletes Phase 2's original tenant-creation flow, Phase 5's approval workflow, and Phase 15's quotation-based billing all at once — see Corrections above.
**Implements:** no `requirement.md` section — every decision below is recorded only via `"requirement.md revisit, confirmed with the user"` comments in the code itself (`app/models/agency.rb`, `app/models/account_switch.rb`, `app/controllers/agency_console/*`, `config/routes.rb`'s own extensive inline commentary).
**Depends on:** Phase 1 (tenant Devise scope, reused as-is for Agency Console login), Phase 2 (`AccountProvisioning`, extended rather than replaced).

### 19.1 `Agency` model and billing
- [x] `Agency` (platform-level, not `TenantScoped`, no `account_id`) — `name`, `subdomain_slug` (same reserved-word/format rules as `Account::RESERVED_SLUGS`, plus a cross-table uniqueness check so an Agency can never silently shadow an Account's slug), `status` (active/suspended), `currency`. → `app/models/agency.rb`, `db/migrate/20260719050000_create_agencies.rb`, `20260719060000_add_subdomain_slug_and_billing_cycle_to_agencies.rb`.
- [x] `billing_cycle` enum, two contracts: `per_event` (`price_per_event`, `events_granted`/`events_used` pool, `Agency#consume_event_slot!` — atomic guarded `UPDATE ... WHERE events_used < events_granted`, raises `Agency::NoEventSlotsRemainingError` on exhaustion) or `annual` (`annual_price`, one upfront lump-sum `Invoice`, `Agency#contract_active?` reads `invoice&.paid?`). → `app/models/agency.rb`.
- [x] `Agency#grant_more!(count)` — Super Admin's "top up the pool" action, additive only (can never set `events_granted` below `events_used`). → `SuperAdmin::AgenciesController#grant_events`.
- [x] `AgencyMembership` — join `User` ↔ `Agency`, one role (`agency_admin`), mirrors `AccountMembership` exactly one tier up (a `platform_staff` user can never hold one). → `app/models/agency_membership.rb`, `db/migrate/20260719050100_create_agency_memberships.rb`.
- [x] `Account#agency` (nullable — a legacy pre-pivot Account has none, left alone rather than backfilled/migrated). → `db/migrate/20260719050200_add_agency_id_to_accounts.rb`.
- [x] `AccountMembership` role set collapsed from 4 to 2 (`owner`/`event_manager` → `event_admin`; `checkin_staff`/`finance_readonly` → `admin_staff`) to make room for the new `agency_admin` tier above — confirmed as a real merge of already-identical Pundit grants, not a behavior change (every policy already checked the two old roles as one unit). → `app/models/account_membership.rb`, data-migrating `db/migrate/20260719050300_remap_account_membership_roles.rb` (raw-integer `UPDATE`, not an ActiveRecord callback, so it's correct regardless of app-code deploy order).

### 19.2 Agency provisioning & tenant creation moves under it
- [x] `AgencyProvisioning` — the only way an Agency comes into existence: Super Admin-only, creates the Agency, its first `agency_admin` User (temp password, forced reset — reuses Phase 1), the `AgencyMembership`, and (for an `annual` agency) its one upfront contract `Invoice`, all in one transaction. → `app/services/agency_provisioning.rb`, `SuperAdmin::AgenciesController#new`/`#create`.
- [x] `AgencyMembershipProvisioning` — adds a *second+* `agency_admin` to an existing Agency (`find_or_initialize_by(email:)`, so an already-platform User is reused rather than duplicated); backfills an `event_admin` `AccountMembership` on every one of that agency's *existing* tenants for the new staff member. → `app/services/agency_membership_provisioning.rb`, `SuperAdmin::AgencyMembershipsController#create`/`#destroy`/`#resend_invite`.
- [x] `AccountProvisioning` extended with an `agency:` kwarg (optional, backward-compatible with the pre-pivot nil case) — when given, links the new Account under it **and** backfills an `event_admin` `AccountMembership` for every existing `agency_admin` of that agency, so the agency's own staff can log into a brand-new tenant the instant it's created, no extra invite step. → `app/services/account_provisioning.rb`.
- [x] `AgencyConsole::AccountsController#new`/`#create` — the *only* place a tenant Account is created now (Super Admin's own `#new`/`#create` removed, see Corrections). Gated on `Current.agency.contract_active?` for an `annual` agency (can't create a single tenant until the upfront contract is paid); a `per_event` agency has no equivalent gate here (its pool only blocks *event* creation, not tenant creation — see 19.3). → `app/controllers/agency_console/accounts_controller.rb`.
- [x] `Event#agency_contract_must_be_active` — the sole gate on event creation now (replaces Phase 5's approval workflow and Phase 15's quotation gate entirely): no agency at all → hard validation error (defensive, covers only a pre-pivot standalone Account); `annual` agency with an unpaid contract → blocked; `per_event` agency with `events_remaining <= 0` → blocked. `Event#consume_agency_slot_if_metered` (`before_create`) does the atomic decrement for a metered agency — a no-op for `annual`. → `app/models/event.rb`.

### 19.3 Agency Console (third console tier)
- [x] `AgencyConsole::BaseController` shares the exact same `Hosting::TenantSubdomainConstraint` and the exact same `:user` Devise scope as the tenant Admin Console — an agency's subdomain is resolved the same way a tenant's is (`TenantResolvable#resolve_tenant!` tries `Account` first, then `Agency`, 404s on neither), and an `agency_admin` is an ordinary `:user`-scope `User` who happens to also hold `AgencyMembership` rows, not a separate Warden scope. → `app/controllers/agency_console/base_controller.rb`, `app/controllers/concerns/tenant_resolvable.rb`.
- [x] `AgencyConsole::DashboardController` — stat row (tenant/event/participant counts, live pulse via `LiveDashboard.agency_pulse`), a "latest 10 tenants, View all →" preview card (full paginated list moved to its own `#index`, per explicit user spec), Paid/Outstanding invoice totals (safe to sum directly in `agency.currency` — an agency's own invoices never span more than one currency), and a real payment-action queue (`awaiting_payment` invoices only — nothing else needs the agency's action). → `app/controllers/agency_console/dashboard_controller.rb`.
- [x] `AgencyConsole::AccountsController#index` — the full paginated tenant list (Pagy, `items: 15`), separate from the dashboard's own 10-row preview. `#switch`/`#suspend`/`#reinstate` — see 19.4/19.5.
- [x] `AgencyConsole::InvoicesController` — every Invoice this agency is responsible for (`Invoice.for_agency`, covering both its own annual-contract row and every per-event row across its tenants), plus the tenant-facing "Mark as Paid" submit flow (`#submit_payment`), reused verbatim from the pre-pivot per-tenant version. → `app/controllers/agency_console/invoices_controller.rb`.
- [x] `AgencyConsole::DirectUploadsController` — this console's own copy of the tenant-scoped direct-upload route (needed for the payment-receipt upload in `#submit_payment`, same reason `Admin::DirectUploadsController` exists). → `app/controllers/agency_console/direct_uploads_controller.rb`.
- [x] `AgencyHelper#agency_nav_items` — Dashboard / Tenants / New Tenant / Invoices, four items. → `app/helpers/agency_helper.rb`.
- [x] Routing: `scope path: "agency", as: "agency"` inside the same `Hosting::TenantSubdomainConstraint` block as the tenant Admin Console (`config/routes.rb`) — `agency_root_path` (`GET /agency`), `agency_accounts_path`, `agency_invoices_path`. Controller module is `AgencyConsole::`, not `Agency::` — `Agency` is already a top-level model class, Zeitwerk can't resolve a name as both.

### 19.4 Agency → Tenant single sign-on (`AccountSwitch`)
- [x] `AccountSwitch` — short-lived (60s), single-use, `SecureRandom.urlsafe_base64(32)` token record (`belongs_to :user`, `belongs_to :account`, `redeemed_at`/`expires_at`). Not `TenantScoped`, platform-level like `Agency`/`Account` themselves — created from a request context where `Current.agency` is set and `Current.account` is nil, so a `TenantScoped` default_scope would raise. → `app/models/account_switch.rb`, `db/migrate/20260719070000_create_account_switches.rb`.
- [x] Mint half: `AgencyConsole::AccountsController#switch` — `Current.agency.accounts.find(params[:id])` **is** the entire authorization boundary (an agency can only ever switch into its own tenant, 404s otherwise), then a genuinely cross-subdomain redirect (`allow_other_host: true`, safe because the target host is built from the already-scoped `account.subdomain_slug`, never from unvalidated request input) straight to the tenant's own `/admin/switch?token=...`.
- [x] Redeem half: `Admin::AccountSwitchesController#redeem` — reachable while signed out (same as `Admin::SessionsController`), a plain `GET` (not `POST`) since it has to work as an ordinary browser redirect target, the same shape Devise's own password-reset link already uses. Signs the user in via `sign_in` under the tenant's own host-only session — never a shared cookie across subdomains, preserving Phase 1's cookie-isolation guarantee. → `app/controllers/admin/account_switches_controller.rb`, route `get "admin/switch", to: "admin/account_switches#redeem", as: :redeem_account_switch`.

- [x] `SuperAdminHelper#super_admin_nav_items` — Dashboard / Agencies / Invoice / Live Pulse. "Event Approvals" and "Quotations" both removed along with the workflow they linked to.

Every read site that used to lean on that default_scope filters explicitly now (`Admin::InvoicesController` deleted outright — invoice management moved to the Agency Console per 19.3 — `SuperAdmin`/`AgencyConsole::InvoicesController` both query without relying on tenant scoping). → migration `20260719060100_relax_invoice_for_agency_contract.rb`.

## Phase 20 — De-partition `ScanEvent`/`Attendance`

**Goal (user request: *"i don't want this kind of partition. keep single database table"*):** reverse Phase 9's own original decision to monthly-range-partition `scan_events`/`attendances` on Postgres native declarative partitioning. Both tables become ordinary single tables, same as every other table in the app.
**Implements:** none (a direct architectural reversal of Phase 9's own `requirement.md §4.10` citation — the requirement doc itself still recommends partitioning; this was a deliberate, explicit override).
**Depends on:** Phase 9 (the tables being de-partitioned).

- [x] `CreateScanEvents`/`CreateAttendances` migrations rewritten from `MonthlyRangePartitioning.create_parent!` to plain `create_table :id, :uuid` — both tables now have an ordinary single-column primary key (`id`) instead of the composite `(id, scanned_at)`/`(id, occurred_at)` a partitioned table required. → `db/migrate/20260712172321_create_scan_events.rb`, `20260712172322_create_attendances.rb` (edited in place, not a new migration — both were still unreleased/local-only, so no data-preserving rollout was needed; confirmed with the user before resetting the local dev DB).
- [x] `attendances.scan_event_id` is now a real foreign key (`t.references :scan_event, foreign_key: true`) — previously impossible, since Postgres can't add an FK against only one column of a partitioned table's composite primary key (the exact limitation the original `Attendance` model comment called out). A genuine correctness improvement that fell out of the reversal, not a separate ask.
- [x] `lib/monthly_range_partitioning.rb` and `app/jobs/partition_maintenance_job.rb` deleted outright — confirmed via grep that `PartitionMaintenanceJob.perform_later` was never actually called anywhere (it was a known, flagged gap in Phase 9/10's own DoD notes), so this is a clean removal, not an orphaned scheduled job left running.
- [x] `ScanEvent#primary_key = "id"`/`Attendance#primary_key = "id"` overrides removed from both models — no longer needed once the DB-level primary key is a plain single column again.
- [x] Every `MonthlyRangePartitioning.ensure_partitions!` call removed from specs (3 call sites — `spec/requests/admin_events_spec.rb`, `spec/requests/admin_scan_events_spec.rb`, `spec/jobs/event_scheduler_job_spec.rb` — each was pre-provisioning a partition before a frozen-time write; no longer needed against a plain table).
- [x] `config/application.rb`'s `schema_format = :sql` comment corrected — it was attributed entirely to partitioning (`PARTITION BY`/child partitions), which schema.rb genuinely can't represent, but the *actual*, still-current reason `:sql` format stays required is Postgres Row Level Security (`ENABLE ROW LEVEL SECURITY`/`CREATE POLICY`, used by 32 other migrations) — schema.rb can't represent that either, and would still force `:sql` format even with partitioning gone.

### Definition of Done
- [x] `bin/rails db:migrate` on a fresh empty DB produces plain (non-partitioned) `scan_events`/`attendances` tables with RLS still enabled — verified directly against Postgres (`\d scan_events` shows a plain `Table`, not `Partitioned table`, `scan_events_pkey` is a single-column `btree (id)`, `POLICY "tenant_isolation"` still present).
- [x] Full suite green: **876 examples, 0 failures**, Rubocop clean on every touched file.
- [x] Local dev/test databases reset and re-migrated to the corrected schema (required stopping the running `puma`/`sidekiq` processes and terminating an idle TablePlus connection first — all confirmed with the user before doing so).
